# dd_idea — 蛋白間配列アラインメントとポケットマッピングの組み合わせ: UniProtアクセッション2つ以上から、活性部位の相違点のヒントへ

2つ以上の**異なる**蛋白（パラログ、オフターゲット、キナーゼファミリー等）の
UniProtアクセッションを指定すると、正準配列同士をアラインメントし、参照とする
1蛋白のdruggableなポケットを他の全蛋白それぞれの残基番号系へマッピングし、
AlphaFold DBモデルを重ね合わせて、活性部位が実際どこでどう異なるかを
色分けして視覚化する。単一のシード蛋白だけを指定した場合には、類似蛋白の
候補を自動提案することもできる（Pfam/InterProファミリー所属＋配列同一性
ランキング）——事前にアクセッションのリストを用意する必要はない。特定の
ターゲットファミリーに依存しない再利用可能なパッケージとして設計されている。
下記の実例（ヒトCDK20/CDK2/MAK）は実際の創薬探索セッションから生まれた
ものだが、任意のUniProtアクセッションの組み合わせで動作する。

これは[`dd_seqalign`](https://github.com/rhara/dd_seqalign)（**同一蛋白**の
全既知構造をその蛋白自身の正準配列と比較する）の「異なる蛋白同士」版に
あたる。ここには複数構造で共有される単一の正準配列というものが存在せず
——各蛋白がそれぞれ自分自身の正準配列を持つ——そのため、固定された正準
位置の集合ではなく、参照蛋白の「ポケット」こそが他蛋白へ変換される対象になる。

- **Fetch (`dd_idea-fetch`)**: 各アクセッションについて正準配列＋
  AlphaFold DBモデルを取得する（再実行時、既にディスクにあるものは
  スキップ）、さらに（`--no-pdb-overlay`が無ければ）各蛋白の実RCSB構造の
  探索・選定も行う（下記「実構造オーバーレイ」参照）——これらは全て正真正銘
  fetch段階のネットワーク処理であり、全てキャッシュされ、`manifest.json`に
  記録される。`--discover SEED_ACC` を使うと、取得の代わりに1つのシード
  アクセッションについて類似蛋白の候補を提案し（下記「類似蛋白の自動探索」
  参照）、`candidates.json` に書き出す——これはあくまで提案であり、実際に
  使いたいアクセッションを選んで通常の `dd_idea-fetch` 呼び出しに渡す。
- **Align (`dd_idea-align`)**: 参照蛋白を1つ選び（デフォルトは最初に
  fetchしたもの）、その AlphaFold モデルに対して `fpocket` を実行して
  druggableなポケットを検出し、他の各蛋白の正準配列を参照の正準配列に
  ペアワイズアラインメントする（Biopythonの`PairwiseAligner`、BLOSUM62、
  global——下記「globalである理由（glocalではなく）」参照）。参照の
  ポケット裏打ち残基を、各蛋白自身の残基番号系へマッピングする。マッピング
  された各位置は `identical` / `conservative` / `non-conservative` / `gap`
  に分類される（BLOSUM62のスコアの符号を使用、独自に保守する置換基準表
  ではない——`dd_idea/sequence.py`参照）。その後、各蛋白のAlphaFoldモデル
  と、`dd_idea-fetch`が既に選定した実構造それぞれを、PyMOLの`cealign`
  （トポロジーのみに基づく全鎖構造重ね合わせ、共有の残基番号系は不要）で
  参照へ重ね合わせる——`dd_seqalign`のもう一方のフィットモードである
  `pair_fit`には、蛋白間で成立する対応物がなく、ここには引き継がれていない。
  このステップ自体はRCSBへのネットワークアクセスを一切行わない——align
  を再実行しても（`--reference`/`--pocket-rank`を変えて試す場合など）、
  fetchが既にキャッシュしたものを再利用するだけである。
- **Run (`dd_idea-run`)**: fetch + align を1ステップで（明示的な
  アクセッションリストのみ対応——候補提案が欲しい場合は`--discover`を
  別途先に実行すること）。
- **App (`streamlit run app.py -- --report-dir DIR`)**: 4タブ構成 --
  Overview（蛋白ごとの長さ／参照との同一性％／RMSDの表）、Active-site
  comparison（ポケット残基マッピング表、保存度で色分け）、Structure
  overlay（今は無き`dd_viewer`プロジェクトから絞り込んでvendorした
  二重バッファ`view3d`コンポーネント——各蛋白のAlphaFoldモデルを
  重ね合わせ、それぞれ別の色で表示し、
  参照ポケット残基を各構造上にハイライト、各ポケット残基に任意でテキスト
  ラベルを表示——ハイライト・ラベルともサイドバーから個別にON/OFF可能で、
  表示する蛋白の選択とは独立），Candidates（レポート
  ディレクトリに`candidates.json`が存在する場合のみ表示——事前の
  `--discover`実行結果のランキング）。ある蛋白が持つ実RCSB構造（下記
  「実構造オーバーレイ」参照）は、それぞれ専用のチェックボックスを持ち、
  サイドバー「Proteins to show」内でその蛋白自身のチェックボックスの下に
  ネストして表示される——その蛋白のAlphaFoldモデルとも、他の実構造とも
  独立にON/OFFできる。表示された実構造は不透明のカートゥーンとして描画され、
  結合リガンドはそれぞれ固有の色でスティック表示される（3Dビュー上部の
  キャプションの色見本でどの色がどのPDB ID/リガンドかを確認できる）。

## 実例: CDK20 vs. CDK2 vs. MAK

```bash
dd_idea-run Q8IZL9 P24941 P20794 -o CDK20_inhibition/cross_protein_comparison --reference Q8IZL9
streamlit run app.py -- --report-dir CDK20_inhibition/cross_protein_comparison
```

ヒトCDK20（Q8IZL9）には実験構造（PDB）が1件も無く、CDK2（P24941）は512件、
MAK（P20794）は0件——まさに「PDB構造がたまたま存在するものではなく、常に
AlphaFoldモデル同士を比較する」方針（下記「常にAlphaFoldである理由」参照）
が意味を持つ状況である。ここでは他に方法がなく、3者を公平に比較すること
自体ができない。CDK20との配列同一性: CDK2 44.1%、MAK 36.5%。本ツールが
自動検出した最上位ポケット（24残基、druggability 0.65）は、参照位置14と
131を、CDK2・MAK**両方**において non-conservative（非保存的）と判定した
——CDK20を区別しうる候補ポケット残基である。加えて位置305はCDK2側で
純粋なgap（CDK2の正準配列は298残基とCDK20の346残基より単純に短いため）。
`CDK20_inhibition/cross_protein_comparison/`（取得済み配列・構造、
`report.json`、重ね合わせ済み座標）は実例としてそのままコミットされて
いる——本ツール自身の検証実行がきっかけとなった、進行中のCDK20阻害薬
開発プロジェクトの一部でもある。詳細は`CDK20_inhibition/README.md`参照。
CDK2には実際に
リガンドが結合したRCSB構造が3件（`6Q4G`/`HJK`解像度0.98Å、`6Q49`/`HGQ`
1.00Å、`6Q4H`/`HGH`1.00Å）もあり、実構造オーバーレイ機能（下記参照）が
自動的にこれらを選び、それぞれ固有の色の不透明カートゥーンとしてCDK2の
AlphaFoldモデルに重ねて表示する——参照からマッピングされたポケット残基が、
3種類の実際に異なるリガンドが結合している位置とどう一致するかを確認できる。

## 類似蛋白の自動探索

`--discover`は、シード蛋白のUniProtエントリに紐づくPfam/InterProファミリー
のうち、メンバーが2つ以上ある**最小の**ものを選ぶ（デフォルトは同一生物種
限定、`--any-organism`で解除可能）。CDK20の場合これはInterPro
`IPR050108`（「CDK」ファミリー、ヒト26メンバー）に解決される——もっと
広い`Pfam:PF00069`（「Pkinase」、344メンバー、「とにかくキナーゼドメインを
持つ」というだけの粒度）でも、狭すぎる`IPR048002`（「CDK20-like_STKc」、
1メンバー=シード自身のみ）でもない。各メンバーの正準配列を取得し、
シードとのグローバルBLOSUM62同一性％でランキングする。

**既知の制約**: これはシードと**同じ**ファミリーに既に分類されている
蛋白しか見つけられない。`dd_idea-fetch --discover Q8IZL9`はCDK2
（同一性44.1%）を上位候補として正しく見つけ出すが、MAKは見つからない
——MAKは本プロジェクト自身の実例（上記）で実際に一緒に検討された、
生物学的に意味のある比較対象であるにもかかわらず、InterPro上でCDK
ファミリーに紐づいていないため、候補プールに一切入らない。CDK20対MAKの
ようなファミリーをまたいだ比較には、明示的なアクセッションリストが必要
——または下記の`dd_idea-search`を使う。ファミリー分類に一切依存しない
ため、MAKも直接見つかる。

## `dd_idea-search`: BLASTベースの入口ツール

類似蛋白を探しポケット検出の入力一式を揃えるための、もう一つの相補的な
方法——Pfam/InterProファミリー所属ではなくSwiss-Prot全体へのBLASTPを
使うため、上記`--discover`の既知の制約を受けない（CDK20に対しMAKを
実際に発見できる——局所同一性35.1%、Homo sapiens限定Swiss-Protヒット中
24位）。UniProtアクセッション、ChEMBLターゲットID(例: `CHEMBL301`)、
または生のアミノ酸配列を直接与えられる——興味のあるChEMBLターゲットや、
まだUniProtに登録されていない配列から始めたい場合に有用。

意図的に1段階ではなく4段階に分けている: まず表を作る(高速・ダウンロード
なし)、それを見てから、軽量な件数のみのシグナル(ChEMBL活性件数、RCSB
構造件数)で表を充実させ、その上で本当に必要なアクセッションだけ実際に
AlphaFoldモデル/PDB構造をダウンロードする。

```bash
dd_idea-search Q8IZL9 -o CDK20_inhibition/pocket_detection
# -> 完全な表(Family/Gene/Organism/Length/%Id/E-value)を表示、ダウンロードなし

dd_idea-search --chembl-activity-all -o CDK20_inhibition/pocket_detection
# -> 各行のChEMBL SINGLE PROTEINターゲットを解決し、pChEMBL値付きのbinding assay
#    活性件数をカウント(dd_chembl自身のQSAR学習データと同じフィルタ)——軽量
#    (件数のみ、ターゲットごとに1リクエスト、実際の活性データはダウンロードしない)
#    なので、--fetch-allと違い--allをデフォルトに使って問題ない

dd_idea-search --pdb-count-all -o CDK20_inhibition/pocket_detection
# -> 各行のアクセッションに紐づくRCSB構造の総数をカウント——アクセッション
#    あたり1回の軽量なsearch-API呼び出しのみ、解像度によるフィルタもダウンロード
#    もなし。--chembl-activity-allと同様に軽量なので、ここでも--allで問題ない。
#    これにより、実際の--fetchが発生する前に--rank(下記)へ本物のテンプレート
#    件数を反映できる——詳細は下記「推奨ワークフロー」参照

dd_idea-search --fetch P24941 P20794 -o CDK20_inhibition/pocket_detection --resolution-cutoff 2.5
# -> AlphaFoldモデル(シードの場合)+その2件の解像度2.5Å以下の全RCSB構造を取得

dd_idea-search --fetch-all -o CDK20_inhibition/pocket_detection --resolution-cutoff 2.5
# -> 同様に表の全行を対象(よく研究されたヒットだと数百件の構造になりうる——下記参照)
```

よく研究されたヒットはRCSBエントリを数百件持つことがある(CDK2だけで
512件、うち2.5Å以下は約450件)——無差別に`--fetch`するのは通常望ましく
ない。`--fetch-all`に頼らず、短く意図的なリストを`--fetch`すること。
`--chembl-activity`にはそのようなコスト懸念がない(アクセッションあたり
数件の軽量REST呼び出しのみ)ので、`--chembl-activity-all`を最初の
一括確認として使うのは妥当——例えばCDK20の100件のBLASTヒットでは、
ChEMBLカバレッジに3000倍超の開き(GSK3B: 7448件の活性データ、CDK20
自身: 1件)があることが分かり、どのヒットがポケット検出テンプレートとして
有用かつ後続のQSAR研究に足るSARデータを持つか、判断材料になった。

`--fetch`したRCSB構造は`raw_pdb/`直下ではなく`raw_pdb/{gene}/`配下に
置かれる(例: `raw_pdb/CDK2/6Q4G.pdb`)——複数アクセッションを対象にした
`--fetch-all`だと、無関係な数十蛋白から数千ファイルがフラットに並んで
しまうため。行にgene名の記録が無い場合(例: UniProtエントリを持たない
生シーケンスのシード)はアクセッション自身にフォールバックする。複数の
アクセッションのUniProtレコードに紐づくPDBエントリ(例: 複数鎖構造)は、
所属する各geneのサブディレクトリにそれぞれコピーされ、各geneフォルダが
自己完結するようになっている。

### NCBIのWeb UIから手動でBLASTを実行する方法

`dd_idea-search`自身の投入処理(`dd_idea/blast/query.py`)はNCBIのBLAST URL
APIを60秒ごとにポーリングし、状態が変わるたびに出力するが、NCBIが返す
RTOE(推定完了時間)自体はサーバー負荷に依存する大まかな数値に過ぎず
——混雑時には実際には1分もかからず終わる検索でも、数千秒という見積もりが
返ってくることがある。検索が止まっているように感じたら、代わりにNCBIの
Web UIから実行し、その結果を`dd_idea-search`自身が読みに行くのと同じ
キャッシュ場所に置く方が速いことが多く、NCBI自身の進捗ページを直接見ながら
進められる:

1. [NCBI Protein BLAST](https://blast.ncbi.nlm.nih.gov/Blast.cgi?PROGRAM=blastp&PAGE_TYPE=BlastSearch&LINK_LOC=blasthome)を開く。
2. アクセッション(例: `Q8IZL9`)または生の配列をクエリ欄に貼り付ける。
3. Database: **UniProtKB/Swiss-Prot(swissprot)**を選択。
4. "Organism"に**Homo sapiens**を追加——`dd_idea-search`のデフォルトの
   生物種限定と合わせるため(`dd_idea-search --any-organism`を使う予定
   なら省略してよい)。
5. "Algorithm parameters"を開き、**Expect threshold**を`1e-10`に設定
   (`dd_idea-search`のデフォルト`--evalue`と一致させる。**Max target
   sequences**はデフォルトの100のままで`--max-hits`のデフォルトと一致
   する)。`--evalue`/`--max-hits`をデフォルト以外で使うつもりなら、
   ここも合わせて変更すること——`dd_idea-search`はキャッシュされた
   XMLが実際にどのパラメータで生成されたかをチェックしない。
6. **BLAST**をクリックして結果ページを待つ(NCBI自身のページが自動的に
   ポーリング・更新される。割り当てられたRIDは
   `https://blast.ncbi.nlm.nih.gov/Blast.cgi?CMD=Get&FORMAT_OBJECT=SearchInfo&RID=<RID>`
   でいつでも確認できる)。
7. 結果ページで**Download All** → **XML**をクリック。
8. ダウンロードしたファイルを`{out_dir}/raw_blast/blastp_swissprot.xml`
   として保存する——`dd_idea-search`自身がキャッシュに使うディレクトリ名・
   ファイル名と正確に一致させること。
9. 通常通り`dd_idea-search ACCESSION -o {out_dir}`を再実行する——既に
   ファイルがあることを検知し、NCBIへの新規検索投入を完全にスキップして、
   そのままヒットのUniProtメタデータ解決へ進む。

### `--rank`: 4つのシグナルを組み合わせる

同一性・テンプレート・活性データ・familyの情報が揃ったら、`--rank`が
4つ目のモードとして使える——既に収集済みの`hits.json`に対する純粋な
計算のみ(ネットワークアクセスなし)で、4つのシグナルを1つのランキングに
統合する:

```bash
dd_idea-search --rank -o CDK20_inhibition/pocket_detection
```

4つのシグナルは全く異なるスケールを持つ(ChEMBL活性件数は0〜7000超、
RCSBテンプレート件数はヒット集合の約半分がゼロ、%同一性は20〜50%程度の
比較的連続したコンパクトな範囲)ため、`dd_idea.rank`は生の数値を掛け合わせる
のではなく(それだと最もスケールの大きい指標が生物学的意味とは無関係に
支配してしまう)、まず各指標を順序クラスに変換してから組み合わせる:

- **同一性クラス**(1〜5) —— ヒット集合内での`%identity`の分位数ビン
- **テンプレート/活性クラス**(デフォルト1〜20) —— 0は常にそれ単独で
  クラス1(CDK20のヒットの約半分はRCSB構造が0件なので、これを最下位
  分位数に含めると実際の非ゼロ件数が1〜2クラスに圧縮されてしまう)。
  非ゼロ値は他の非ゼロ値の中での分位数でビン分けする。同一性/family
  よりも粒度を広く取っている(`count_classes`、`dd_idea.rank.rank_hits`
  参照)のは、この2つのシグナルが桁違いに広い生の範囲を持つため(テンプレート
  数百件、活性データ数千件)——5クラスのままだと、実際には意味のある差が
  ある件数同士(例: 522件と28件のテンプレート)が両方ともクラス5に丸め
  られてしまい、頻繁に同点になっていた。テンプレート件数自体は、その
  行に`--fetch`/`--fetch-all`による正確な(解像度でフィルタ済みの)件数が
  あればそれを優先し、無ければより軽量な`--pdb-count`/`--pdb-count-all`の
  (解像度フィルタなしの)総数にフォールバックする——このフォールバックが
  何のためにあるかは下記「推奨ワークフロー」参照
- **Familyクラス**(1〜5) —— ヒットのUniProt family階層(superfamily→
  family→subfamily)がシードとどこまで深く一致するか: 最上位から辿って
  subfamilyまで完全一致すれば最上位クラス、family一致・subfamily不一致
  なら1つ下、superfamilyのみ一致なら中間クラス。テンプレート/活性とは
  違い5クラスのまま——一致の深さは実質4パターンしかない真に離散的な
  シグナルなので、これ以上細分化しても情報は増えず、同点を割るだけになる

複合スコアは4クラスの積(デフォルト設定で1〜10000 = 5×20×20×5)。CDK20の
場合、これにより「実データも豊富な近縁パラログ」(CDK2/CDK7/CDK9——いずれも
同じCDC2/CDKX subfamily、高い同一性、数十〜数百件のテンプレート、数百〜
数千件の活性データ)と、「よく研究されているが遠縁のキナーゼ」(MAPK14/
MAPK1/DYRK1A——ChEMBL/テンプレート件数は膨大だがCMGCの別subfamily)、
「構造的に興味深いがデータが乏しい」(例: MAK——familyは一致し本プロジェクトの
最初の実例にも登場したが、ChEMBL活性はわずか13件、RCSB構造は0件)を
正しく区別できている——さらに、テンプレート/活性の粒度を広げたことで、
CDK2(テンプレート522件、活性3015件)とCDK9(テンプレート28件、活性2051件)
も同点にせず区別できるようになった。

### 推奨ワークフロー: 取得の前にランキングする

RCSBテンプレート件数は`--rank`の4シグナルの1つだが、本当にコストの高い
処理(`--fetch`/`--fetch-all`)の完了を待たなければならないとしたら
おかしい——それではランキング自体が循環してしまう: 「構造取得に値する
アクセッションが分かって初めて構造取得ができる」という状態になって
しまうからだ。`--pdb-count`/`--pdb-count-all`はこの循環を断ち切る:
アクセッションあたり1回の軽量なRCSB search-API呼び出し(解像度フィルタ
なし、ダウンロードなし)だけで、実際に何もダウンロードする**前**に
`--rank`へ本物の(ただし解像度未フィルタの)テンプレート件数シグナルを
与えられる——これはChEMBLカバレッジに対して`--chembl-activity`/
`--chembl-activity-all`が既に果たしている役割と全く同じである。
`--rank`は、ある行に`--fetch`/`--fetch-all`による正確な件数が記録されて
いればそれを優先し、無い場合にのみ`--pdb-count`の総数にフォールバック
する(`dd_idea.rank._template_count`参照)——そのため、実際にfetchした後に
再ランキングすれば、fetch済みの行だけ数値が精密化され、全件を先に
fetchしておく必要はない。

```bash
dd_idea-search Q8IZL9 -o CDK20_inhibition/pocket_detection
dd_idea-search --chembl-activity-all -o CDK20_inhibition/pocket_detection
dd_idea-search --pdb-count-all -o CDK20_inhibition/pocket_detection
dd_idea-search --rank -o CDK20_inhibition/pocket_detection --summary-format markdown
# -> hits_ranked.mdを確認する——同一性・ChEMBL活性件数・実際のRCSB
#    テンプレート件数まで全て既に反映されており、それでいて実際の構造
#    ダウンロード(AlphaFoldもPDBも)は一切発生していない

dd_idea-search --fetch P24941 P50613 P50750 Q00535 P06493 Q9NYV4 Q00534 Q16539 P11802 P28482 \
    P49841 Q13627 P49336 Q13164 Q92630 P53779 P45983 P45984 P68400 P49759 \
    -o CDK20_inhibition/pocket_detection
# -> hits_ranked.mdから読み取った上位20件のアクセッションを手で貼り付け。
#    --fetch-all(全ヒット対象)ではなくこの20件だけ、実際の(解像度で
#    フィルタ済みの)構造を取得する(CDK20の99件を--pdb-count-allすると
#    1分足らずでRCSB構造2590件(解像度フィルタなし)と判明。比較として
#    --fetch-allで2.0Å以下を全件取得すると数分・420MBかかった——上位20件の
#    ショートリストならその一部で済む)

dd_idea-search --rank -o CDK20_inhibition/pocket_detection
# -> 再実行して、上位20件のテンプレート件数を--pdb-countの未フィルタ総数から
#    --fetchの正確な解像度フィルタ済み件数へ精密化する
```

`--fetch-all`は不要になったわけではない(ターゲットの絞り込みが済んだ後、
最終的に網羅的に取得したい場合など)——ただし最初に使うものではない。

## インストール

Biopython、pandas、numpy、PyMOL（`pymol2`、ライブラリとしてimport可能
——conda-forgeパッケージ名は`pymol-open-source`だが、`pip`/`pip show`
から見える配布物名は`pymol`）、および`fpocket`CLI（conda-forgeのみ、
PyPIには無く、サブプロセスとして呼び出し）が必要。専用のconda env:

```bash
mamba create -n dd_idea -c conda-forge python=3.12 biopython pandas \
    numpy matplotlib py3dmol streamlit pymol-open-source fpocket
conda activate dd_idea

cd dd_idea && pip install --no-deps -e ".[app]"  # [app]でstreamlit/py3Dmolも追加
```

これにより3つのコンソールコマンドがインストールされる: `dd_idea-fetch`、
`dd_idea-align`、`dd_idea-run`。

## 使い方

```bash
dd_idea-fetch --discover Q8IZL9 -o data/discover   # 任意: 1シードから候補提案
dd_idea-run Q8IZL9 P24941 P20794 -o data --reference Q8IZL9
streamlit run app.py -- --report-dir data
```

`--reference`（デフォルト: 最初にfetchしたアクセッション）は、どの蛋白の
ポケットを検出し、他の全てをどれに重ね合わせるかを決める。`--pocket-rank`
（デフォルト1、fpocketのDruggability Score順位で最上位）は、最上位ポケットが
目的の部位でない場合に別のポケットを選ぶ。

`dd_idea-fetch`/`-run`はさらに以下を受け付ける（何を制御するかは下記
「実構造オーバーレイ」参照）: `--no-pdb-overlay`で実RCSB構造の探索自体を
スキップできる（従来通りAlphaFoldモデルのみでfetch/alignする）。
`--pdb-max-structures N`（デフォルト3）は、蛋白ごとに保持する「リガンドが
異なる実構造」の最大件数。`--pdb-resolution-cutoff N`（デフォルト2.0、
単位Å）は、この解像度より悪い候補（NMR構造などそもそも解像度が報告されて
いないものも含む）をダウンロードすらせずに除外する——リガンド結合の探索・
最良解像度フォールバックの両方に適用される。`--pdb-scan-cap N`
（デフォルト25）は、解像度順にランク付けした候補のうち、結合リガンドの
有無を確認するために調べる件数の上限——その件数分`--pdb-max-structures`が
見つからなければ最良解像度1件へのフォールバックを諦める前の上限で、構造が
数百件あるターゲットでもネットワーク・ダウンロードコストを抑える。
`dd_idea-align`はこれらのオプションを一切受け付けない——RCSBへの
ネットワークアクセスを自分では一切行わず、`dd_idea-fetch`が既に
キャッシュしたものを再利用するだけだからである（下記参照）。

全コマンドは完了した項目ごとに1行ずつ即座に出力する。`--no-progress`で
これを抑制し、最終サマリーのみ表示できる。

## 実構造オーバーレイ

`dd_idea-fetch`/`-run`は、各蛋白について`--pdb-resolution-cutoff`以上の
解像度を持つ実際のRCSB構造があるかどうかを調べ、あれば`--pdb-max-structures`
件まで、**リガンドの種類ごとに1件**を選定する（本物のリガンドが結合して
いるエントリを優先——水・凍結防止剤・補因子は除く。同じリガンドが複数候補
に現れた場合はその中で最良解像度のものが勝つ）、リガンド結合エントリが
1件も見つからなければ（同じ解像度カットオフの対象内で）最良解像度の1件
だけへフォールバックする——選定結果は`manifest.json`に記録される。この
選定処理は正真正銘fetch段階のネットワーク処理であり、正準配列/AlphaFold
モデルのダウンロードと全く同じようにキャッシュされる。**既にfetch済みの
蛋白について、もっと実構造が欲しい場合**は、同じ`-o`ディレクトリに対して
`--pdb-max-structures`を大きくして（必要なら`--pdb-resolution-cutoff`も
緩めて）`dd_idea-fetch`を再実行するだけでよい——既にダウンロード済みの
エントリは再利用され、要求を満たすために必要な追加分だけが新たに取得
される:

```bash
dd_idea-fetch Q8IZL9 P24941 P20794 -o data --pdb-max-structures 6
```

`dd_idea-align`は、既に選定済みの各実構造を、その蛋白のAlphaFoldモデル
に重ねて参照へ超重ね合わせる——自分ではRCSBへのアクセスを一切行わない。
これはあくまで**追加の**可視化レイヤーであり、ポケット検出や蛋白間の配列／
ポケットマッピングは、これまで通り無条件でAlphaFoldモデルに固定された
ままである——下記「常にAlphaFoldである理由」は実際の比較についてはそのまま
成立する。`report.json`には蛋白ごとの選定結果がリストとして記録される:
`pdb_structures: [{pdb_id, resolution, ligand_resname, ...}, ...]`。RCSB
構造が1件も無い蛋白（CDK20自身など）は単に`pdb_structures: []`となる。

**表示して確認する**: コミット済みのCDK20/CDK2/MAK実例には既にCDK2の実
構造オーバーレイ——リガンドの異なる3件——が含まれている——追加のfetchは
不要で、そのままアプリを起動すればよい:

```bash
streamlit run app.py -- --report-dir CDK20_inhibition/cross_protein_comparison
```

**Structure overlay**タブを開く。サイドバー「Proteins to show」内で、
各実構造は専用のチェックボックスを、その蛋白自身のチェックボックスの下に
インデントして持ち（例: `P24941`の下にインデントされた`6Q4G (HJK,
0.98Å)`）、その蛋白自身のAlphaFoldチェックボックスとも、他の実構造とも
独立にON/OFFできる——一度に比較したい件数だけチェックすればよい。表示
された各実構造は不透明のカートゥーンとして、それぞれ固有の色で描画される
（カートゥーンと結合リガンドのスティック表示の両方——黒・グレーを含まない
パレットから循環選択される。無彩色の色は見分けづらいため）。
3Dビュー上部のキャプションの色見本でどの色がどのPDB ID/リガンドかを確認
できる——参照からマッピングされたポケット残基が、複数の実際に異なる既知
リガンドが結合している位置とどう一致するかを直接目視で確認できる。
キャプションには、RCSB構造が1件も無かった蛋白（CDK20やMAK）も示される。
3Dビューのカメラ位置は、どのチェックボックスを切り替えても保持される
（「Reset view」を押した時だけリセットされる）——比較の途中で表示する
構造を切り替えても、視点が乱されることはない。

## 設計メモ

- **姉妹`dd_*`プロジェクトからはimportでなくvendor（コピー）**:
  `pdbio.py`（HETATM分類）、`pocket.py`（`fpocket`ラッパー）、
  `structalign.py`の`cealign`部分は、（`dd_seqalign`から、さらに元を
  辿れば`dd_prep`/`dd_afpocket`から）そのままコピーして持ち込んでおり、
  インストール時に姉妹パッケージへ依存しない——`dd_*`ファミリー全体で
  確立された慣例であり、各envのパッケージバージョンが実際には既に
  乖離している現状でも、各プロジェクトが単体でインストール可能な状態を
  保てる。`dd_seqalign`のもう一つの構造フィットモードである、既知の
  1対1残基対応に基づく`cmd.pair_fit`は引き継がれていない——両側が既に
  同じ正準UniProt位置を共有していることが前提であり、これは**同一蛋白**の
  複数構造の場合にしか成立しないため。`pdbstruct.py`（上記の実構造
  オーバーレイ探索）はさらに、`dd_seqalign.fetch`のRCSB検索/ダウンロード
  関数と、`dd_seqalign.sequence`の鎖ごとの正準配列アラインメント
  （`pick_target_chain`、`align_to_canonical`）をvendorしている——実PDB
  エントリに結合パートナー（サイクリンなど）が含まれる場合に正しい鎖を
  選ぶこと、および参照ポケット残基（正準番号系で定義）をそのエントリ
  自身の非正準な残基番号へ変換してラベル配置することに必要で、
  AlphaFold限定の経路ではこれまで一切不要だった処理（下記参照）。
- **常にAlphaFoldである理由（PDB構造ではなく）——比較そのものについて**:
  「利用可能な最良のPDB構造、無ければAlphaFold」という方針だと、蛋白ごとに
  構造の質やリガンド結合状態がバラバラになってしまう（上記CDK20/CDK2/MAKの
  数字を参照——実構造0件・512件・0件）。これでは「この活性部位の違いは
  本物の生物学的差異なのか、それとも実共結晶とapoモデルを比較しているだけの
  アーティファクトなのか」が区別できなくなる。AlphaFoldモデルに統一する
  ことで、ポケット検出と蛋白間の配列／ポケットマッピングを公平
  （apples-to-apples）に保てる——これは上記の実構造オーバーレイを追加した
  後も無条件で成立している。オーバーレイは表示専用であり、どちらにも
  フィードバックされない。副次的な利点: AlphaFoldモデルの残基番号は常に
  その蛋白自身の正準UniProt配列位置と完全に一致する（ギャップ・挿入・
  別番号系が一切ない）ため、参照上で検出したポケット残基は、蛋白間配列
  アラインメント自体が使っているのと全く同じ座標系で既に表現されている
  ——実PDB構造の非正準な番号付けを扱うために存在する
  `dd_seqalign.activesite`とは異なり、この部分では別途の構造番号系との
  往復変換が一切不要である。ただし`pdbstruct.py`自身のオーバーレイ用
  ラベル配置は、実構造の番号系へ変換する必要があるため、まさにその
  往復変換を必要とする。
- **glocalでなくglobalな配列アラインメントである理由**: `dd_seqalign`は
  **同一蛋白**の断片／アイソフォームを、その蛋白自身の正準配列に対して
  自由末端ギャップ（glocal）でアラインメントする——共結晶断片がN末/C末を
  欠いていても、それは本物の生物学的差異ではないため。異なるパラログ
  同士は本当にN末/C末の伸長が異なりうる（CDK20の346残基とMAKの623残基を
  比較）ため、glocalアラインメントだと末端の本物の配列差異を暗黙に
  無視してしまう——そのため通常のglobalアラインメントを使用している。
- **保存度はBLOSUM62スコアの符号で判定、独自の置換基準グループ表では
  ない**: ある置換がBLOSUM62スコア正なら「conservative」（例:
  Phe→Tyrは+3）、そうでなければ「non-conservative」（例: Phe→Aspは-3）
  ——アラインメント自体が使っているのと同じスコア行列を再利用しており、
  それと食い違う可能性のある別の物理化学的グルーピングを独自に保守する
  ことを避けている。
- **rdkitもC++もGPUも無し**: 本プロジェクトには低分子を扱う処理が一切
  無い（`dd_seqalign`は今は無き`dd_viewer`プロジェクトをまるごとvendor
  しているため、dd_ideaが一切使わない単一受容体シーンビルダーのために
  `rdkit`まで引き連れてしまっているのとは対照的——ここではその同じ
  （既に廃止済みの）`dd_viewer`プロジェクトから、単純な文字列パッチと
  コンポーネント部分だけを直接絞り込んでvendorしている、
  `dd_idea/viewer3d/__init__.py`参照）。本プロジェクト自身の処理は
  RESTフェッチと配列アラインメント（どちらも軽量）で、それより重い処理は
  既にコンパイル済みのツール（`fpocket`、PyMOLのC++コア）に任せている
  ——`dd_afpocket`のMDサンプリングや`dd_docking`のドッキング実行のような、
  本プロジェクト自身の高速化すべきホットな数値ループもGPU並列化可能な
  処理も存在しない。

## 既知の制約

- **Pfam/InterProベースの自動探索は同一ファミリーの蛋白しか見つけられ
  ない**（上記「類似蛋白の自動探索」参照）——機能的に関連していても分類が
  異なる蛋白（CDK20/CDK2に対するMAKなど）には明示的なアクセッション
  リストが必要。
- **fpocketの自動検出ポケットは、側鎖ではなく主鎖原子のみで寄与している
  残基を見落とすことがある**——例えば側鎖がATPポケットから外を向いて
  いても、主鎖のN-H/C=Oがアデニンと水素結合しているキナーゼのヒンジ残基
  など。そのような残基は、文献上既知の選択性決定residueであっても、
  `--pocket-rank 1`の裏打ち残基リストに一切現れないことがありうる。
  `--pocket-rank`で別のfpocketランクのポケットを試すことはできるが、
  fpocketの幾何学的ポケット検出が選ばなかった特定の残基を強制的に比較へ
  含める方法は現状無い。
- **常にAlphaFoldモデルであり、実際のリガンド結合コンフォメーションでは
  ない**（比較そのものについては上記「設計メモ」で述べた意図的な選択）
  ——リガンド結合に伴う誘導適合（induced fit）によるポケット変化はこの
  比較の対象外である。実構造オーバーレイは表示専用。
- **実構造オーバーレイは蛋白ごとに`--pdb-max-structures`件まで**
  （デフォルト3、リガンドの種類ごとに1件）——`--pdb-resolution-cutoff`
  以上の解像度を持つ`--pdb-scan-cap`件の候補の中からリガンド結合
  エントリを優先するが、CDK2の512件のように構造が大量にあるターゲット
  でも全件は走査しない。そのため走査範囲外にリガンド結合エントリが
  存在すれば見逃す可能性があり、そのターゲットがこれまでに共結晶化
  された全リガンドを表示するわけではなく、選ばれた分だけである。重複
  判定は厳密なリガンドresnameの一致によるため、まれに同じ3文字の
  RCSBコードを共有する別々のリガンドがあれば、同一のものとして扱われて
  しまう。解像度カットオフを満たす構造が1件も無いターゲットは、RCSB
  構造が1件も無いターゲットと同様、実構造オーバーレイ自体が表示され
  ない。
