# dd_compare — 蛋白間配列アラインメントとポケットマッピングの組み合わせ: UniProtアクセッション2つ以上から、活性部位の相違点のヒントへ

2つ以上の**異なる**蛋白（パラログ、オフターゲット、キナーゼファミリー等）の
UniProtアクセッションを指定すると、正準配列同士をアラインメントし、参照とする
1蛋白のdruggableなポケットを他の全蛋白それぞれの残基番号系へマッピングし、
AlphaFold DBモデルを重ね合わせて、活性部位が実際どこでどう異なるかを
色分けして視覚化する。単一のシード蛋白だけを指定した場合には、類似蛋白の
候補を自動提案することもできる（Pfam/InterProファミリー所属＋配列同一性
ランキング）——事前にアクセッションのリストを用意する必要はない。特定の
ターゲットファミリーに依存しない再利用可能なパッケージとして設計されている。
下記の実例（ヒトCDK20/CDK2/MAK）は実際の創薬探索セッションから生まれた
ものだが、任意のUniProtアクセッションの組み合わせで動作する。

これは[`dd_seqalign`](https://github.com/rhara/dd_seqalign)（**同一蛋白**の
全既知構造をその蛋白自身の正準配列と比較する）の「異なる蛋白同士」版に
あたる。ここには複数構造で共有される単一の正準配列というものが存在せず
——各蛋白がそれぞれ自分自身の正準配列を持つ——そのため、固定された正準
位置の集合ではなく、参照蛋白の「ポケット」こそが他蛋白へ変換される対象になる。

- **Fetch (`dd_compare-fetch`)**: 各アクセッションについて正準配列＋
  AlphaFold DBモデルを取得する（再実行時、既にディスクにあるものは
  スキップ）。`--discover SEED_ACC` を使うと、取得の代わりに1つのシード
  アクセッションについて類似蛋白の候補を提案し（下記「類似蛋白の自動探索」
  参照）、`candidates.json` に書き出す——これはあくまで提案であり、実際に
  使いたいアクセッションを選んで通常の `dd_compare-fetch` 呼び出しに渡す。
- **Align (`dd_compare-align`)**: 参照蛋白を1つ選び（デフォルトは最初に
  fetchしたもの）、その AlphaFold モデルに対して `fpocket` を実行して
  druggableなポケットを検出し、他の各蛋白の正準配列を参照の正準配列に
  ペアワイズアラインメントする（Biopythonの`PairwiseAligner`、BLOSUM62、
  global——下記「globalである理由（glocalではなく）」参照）。参照の
  ポケット裏打ち残基を、各蛋白自身の残基番号系へマッピングする。マッピング
  された各位置は `identical` / `conservative` / `non-conservative` / `gap`
  に分類される（BLOSUM62のスコアの符号を使用、独自に保守する置換基準表
  ではない——`dd_compare/sequence.py`参照）。その後、各蛋白のAlphaFold
  モデルを、PyMOLの`cealign`（トポロジーのみに基づく全鎖構造重ね合わせ、
  共有の残基番号系は不要）で参照へ重ね合わせる——`dd_seqalign`のもう一方の
  フィットモードである`pair_fit`には、蛋白間で成立する対応物がなく、
  ここには引き継がれていない。
- **Run (`dd_compare-run`)**: fetch + align を1ステップで（明示的な
  アクセッションリストのみ対応——候補提案が欲しい場合は`--discover`を
  別途先に実行すること）。
- **App (`streamlit run app.py -- --report-dir DIR`)**: 4タブ構成 --
  Overview（蛋白ごとの長さ／参照との同一性％／RMSDの表）、Active-site
  comparison（ポケット残基マッピング表、保存度で色分け）、Structure
  overlay（`dd_viewer`の二重バッファ`view3d`コンポーネントを絞り込んで
  vendor——各蛋白のAlphaFoldモデルを重ね合わせ、それぞれ別の色で表示し、
  参照ポケット残基を各構造上にハイライト、各ポケット残基に任意でテキスト
  ラベルを表示——ハイライト・ラベルともサイドバーから個別にON/OFF可能で、
  表示する蛋白の選択とは独立），Candidates（レポート
  ディレクトリに`candidates.json`が存在する場合のみ表示——事前の
  `--discover`実行結果のランキング）。ある蛋白が持つ実RCSB構造（下記
  「実構造オーバーレイ」参照）は、それぞれ専用のチェックボックスを持ち、
  サイドバー「Proteins to show」内でその蛋白自身のチェックボックスの下に
  ネストして表示される——その蛋白のAlphaFoldモデルとも、他の実構造とも
  独立にON/OFFできる。表示された実構造は半透明レイヤーとして描画され、
  結合リガンドはそれぞれ固有の色でスティック表示される（3Dビュー上部の
  キャプションの色見本でどの色がどのPDB ID/リガンドかを確認できる）。

## 実例: CDK20 vs. CDK2 vs. MAK

```bash
dd_compare-run Q8IZL9 P24941 P20794 -o data/example_cdk20_cdk2_mak --reference Q8IZL9
streamlit run app.py -- --report-dir data/example_cdk20_cdk2_mak
```

ヒトCDK20（Q8IZL9）には実験構造（PDB）が1件も無く、CDK2（P24941）は512件、
MAK（P20794）は0件——まさに「PDB構造がたまたま存在するものではなく、常に
AlphaFoldモデル同士を比較する」方針（下記「常にAlphaFoldである理由」参照）
が意味を持つ状況である。ここでは他に方法がなく、3者を公平に比較すること
自体ができない。CDK20との配列同一性: CDK2 44.1%、MAK 36.5%。本ツールが
自動検出した最上位ポケット（24残基、druggability 0.65）は、参照位置14と
131を、CDK2・MAK**両方**において non-conservative（非保存的）と判定した
——CDK20を区別しうる候補ポケット残基である。加えて位置305はCDK2側で
純粋なgap（CDK2の正準配列は298残基とCDK20の346残基より単純に短いため）。
`data/example_cdk20_cdk2_mak/`（取得済み配列・構造、`report.json`、重ね
合わせ済み座標）は実例としてそのままコミットされている。CDK2には実際に
リガンドが結合したRCSB構造が3件（`6Q4G`/`HJK`解像度0.98Å、`6Q49`/`HGQ`
1.00Å、`6Q4H`/`HGH`1.00Å）もあり、実構造オーバーレイ機能（下記参照）が
自動的にこれらを選び、それぞれ固有の色でCDK2のAlphaFoldモデルに重ねて
半透明レイヤーとして表示する——参照からマッピングされたポケット残基が、
3種類の実際に異なるリガンドが結合している位置とどう一致するかを確認できる。

## 類似蛋白の自動探索

`--discover`は、シード蛋白のUniProtエントリに紐づくPfam/InterProファミリー
のうち、メンバーが2つ以上ある**最小の**ものを選ぶ（デフォルトは同一生物種
限定、`--any-organism`で解除可能）。CDK20の場合これはInterPro
`IPR050108`（「CDK」ファミリー、ヒト26メンバー）に解決される——もっと
広い`Pfam:PF00069`（「Pkinase」、344メンバー、「とにかくキナーゼドメインを
持つ」というだけの粒度）でも、狭すぎる`IPR048002`（「CDK20-like_STKc」、
1メンバー=シード自身のみ）でもない。各メンバーの正準配列を取得し、
シードとのグローバルBLOSUM62同一性％でランキングする。

**既知の制約**: これはシードと**同じ**ファミリーに既に分類されている
蛋白しか見つけられない。`dd_compare-fetch --discover Q8IZL9`はCDK2
（同一性44.1%）を上位候補として正しく見つけ出すが、MAKは見つからない
——MAKは本プロジェクト自身の実例（上記）で実際に一緒に検討された、
生物学的に意味のある比較対象であるにもかかわらず、InterPro上でCDK
ファミリーに紐づいていないため、候補プールに一切入らない。CDK20対MAKの
ようなファミリーをまたいだ比較には、明示的なアクセッションリストが必要。

## インストール

Biopython、pandas、numpy、PyMOL（`pymol2`、ライブラリとしてimport可能
——conda-forgeパッケージ名は`pymol-open-source`だが、`pip`/`pip show`
から見える配布物名は`pymol`）、および`fpocket`CLI（conda-forgeのみ、
PyPIには無く、サブプロセスとして呼び出し）が必要。専用のconda env:

```bash
mamba create -n dd_compare -c conda-forge python=3.12 biopython pandas \
    numpy matplotlib py3dmol streamlit pymol-open-source fpocket
conda activate dd_compare

cd dd_compare && pip install --no-deps -e ".[app]"  # [app]でstreamlit/py3Dmolも追加
```

これにより3つのコンソールコマンドがインストールされる: `dd_compare-fetch`、
`dd_compare-align`、`dd_compare-run`。

## 使い方

```bash
dd_compare-fetch --discover Q8IZL9 -o data/discover   # 任意: 1シードから候補提案
dd_compare-run Q8IZL9 P24941 P20794 -o data --reference Q8IZL9
streamlit run app.py -- --report-dir data
```

`--reference`（デフォルト: 最初にfetchしたアクセッション）は、どの蛋白の
ポケットを検出し、他の全てをどれに重ね合わせるかを決める。`--pocket-rank`
（デフォルト1、fpocketのDruggability Score順位で最上位）は、最上位ポケットが
目的の部位でない場合に別のポケットを選ぶ。`--no-pdb-overlay`で下記の実
RCSB構造の探索をスキップできる（従来通りAlphaFoldモデルのみでfetch/align
する）。`--pdb-max-structures N`（デフォルト3）は、蛋白ごとに保持する
「リガンドが異なる実構造」の最大件数。`--pdb-scan-cap N`（デフォルト25）は、
解像度順にランク付けした候補のうち、結合リガンドの有無を確認するために
実際にダウンロードして調べる件数の上限——その件数分見つからなければ
最良解像度1件へのフォールバックを諦める前の上限で、構造が数百件ある
ターゲットでもネットワーク・ダウンロードコストを抑える。

全コマンドは完了した項目ごとに1行ずつ即座に出力する。`--no-progress`で
これを抑制し、最終サマリーのみ表示できる。

## 実構造オーバーレイ

`dd_compare-align`/`-run`は、各蛋白について実際のRCSB構造があるかどうかも
調べ、あれば`--pdb-max-structures`件まで、**リガンドの種類ごとに1件**
選んで（本物のリガンドが結合しているエントリを優先——水・凍結防止剤・
補因子は除く。同じリガンドが複数候補に現れた場合はその中で最良解像度の
ものが勝つ）、リガンド結合エントリが1件も見つからなければ最良解像度の
1件だけへフォールバックし、それぞれをその蛋白のAlphaFoldモデルに重ねて
参照へ超重ね合わせる。これはあくまで**追加の**可視化レイヤーであり、
ポケット検出や蛋白間の配列／ポケットマッピングは、これまで通り無条件で
AlphaFoldモデルに固定されたままである——下記「常にAlphaFoldである理由」は
実際の比較についてはそのまま成立する。`report.json`には蛋白ごとの選定
結果がリストとして記録される: `pdb_structures: [{pdb_id, resolution,
ligand_resname, ...}, ...]`。RCSB構造が1件も無い蛋白（CDK20自身など）は
単に`pdb_structures: []`となる。

**表示して確認する**: コミット済みのCDK20/CDK2/MAK実例には既にCDK2の実
構造オーバーレイ——リガンドの異なる3件——が含まれている——追加のfetchは
不要で、そのままアプリを起動すればよい:

```bash
streamlit run app.py -- --report-dir data/example_cdk20_cdk2_mak
```

**Structure overlay**タブを開く。サイドバー「Proteins to show」内で、
各実構造は専用のチェックボックスを持ち（例: `P24941`の下に`↳ 6Q4G
(HJK, 0.98Å)`）、その蛋白自身のAlphaFoldチェックボックスとも、他の実構造
とも独立にON/OFFできる——一度に比較したい件数だけチェックすればよい。
表示された各実構造は半透明レイヤーとして描画され、結合リガンドはそれぞれ
固有の色でスティック表示される（3Dビュー上部のキャプションの色見本で
どの色がどのPDB ID/リガンドかを確認できる）——参照からマッピングされた
ポケット残基が、複数の実際に異なる既知リガンドが結合している位置と
どう一致するかを直接目視で確認できる。キャプションには、RCSB構造が
1件も無かった蛋白（CDK20やMAK）も示される。

## 設計メモ

- **姉妹`dd_*`プロジェクトからはimportでなくvendor（コピー）**:
  `pdbio.py`（HETATM分類）、`pocket.py`（`fpocket`ラッパー）、
  `structalign.py`の`cealign`部分は、（`dd_seqalign`から、さらに元を
  辿れば`dd_prep`/`dd_afpocket`から）そのままコピーして持ち込んでおり、
  インストール時に姉妹パッケージへ依存しない——`dd_*`ファミリー全体で
  確立された慣例であり、各envのパッケージバージョンが実際には既に
  乖離している現状でも、各プロジェクトが単体でインストール可能な状態を
  保てる。`dd_seqalign`のもう一つの構造フィットモードである、既知の
  1対1残基対応に基づく`cmd.pair_fit`は引き継がれていない——両側が既に
  同じ正準UniProt位置を共有していることが前提であり、これは**同一蛋白**の
  複数構造の場合にしか成立しないため。`pdbstruct.py`（上記の実構造
  オーバーレイ探索）はさらに、`dd_seqalign.fetch`のRCSB検索/ダウンロード
  関数と、`dd_seqalign.sequence`の鎖ごとの正準配列アラインメント
  （`pick_target_chain`、`align_to_canonical`）をvendorしている——実PDB
  エントリに結合パートナー（サイクリンなど）が含まれる場合に正しい鎖を
  選ぶこと、および参照ポケット残基（正準番号系で定義）をそのエントリ
  自身の非正準な残基番号へ変換してラベル配置することに必要で、
  AlphaFold限定の経路ではこれまで一切不要だった処理（下記参照）。
- **常にAlphaFoldである理由（PDB構造ではなく）——比較そのものについて**:
  「利用可能な最良のPDB構造、無ければAlphaFold」という方針だと、蛋白ごとに
  構造の質やリガンド結合状態がバラバラになってしまう（上記CDK20/CDK2/MAKの
  数字を参照——実構造0件・512件・0件）。これでは「この活性部位の違いは
  本物の生物学的差異なのか、それとも実共結晶とapoモデルを比較しているだけの
  アーティファクトなのか」が区別できなくなる。AlphaFoldモデルに統一する
  ことで、ポケット検出と蛋白間の配列／ポケットマッピングを公平
  （apples-to-apples）に保てる——これは上記の実構造オーバーレイを追加した
  後も無条件で成立している。オーバーレイは表示専用であり、どちらにも
  フィードバックされない。副次的な利点: AlphaFoldモデルの残基番号は常に
  その蛋白自身の正準UniProt配列位置と完全に一致する（ギャップ・挿入・
  別番号系が一切ない）ため、参照上で検出したポケット残基は、蛋白間配列
  アラインメント自体が使っているのと全く同じ座標系で既に表現されている
  ——実PDB構造の非正準な番号付けを扱うために存在する
  `dd_seqalign.activesite`とは異なり、この部分では別途の構造番号系との
  往復変換が一切不要である。ただし`pdbstruct.py`自身のオーバーレイ用
  ラベル配置は、実構造の番号系へ変換する必要があるため、まさにその
  往復変換を必要とする。
- **glocalでなくglobalな配列アラインメントである理由**: `dd_seqalign`は
  **同一蛋白**の断片／アイソフォームを、その蛋白自身の正準配列に対して
  自由末端ギャップ（glocal）でアラインメントする——共結晶断片がN末/C末を
  欠いていても、それは本物の生物学的差異ではないため。異なるパラログ
  同士は本当にN末/C末の伸長が異なりうる（CDK20の346残基とMAKの623残基を
  比較）ため、glocalアラインメントだと末端の本物の配列差異を暗黙に
  無視してしまう——そのため通常のglobalアラインメントを使用している。
- **保存度はBLOSUM62スコアの符号で判定、独自の置換基準グループ表では
  ない**: ある置換がBLOSUM62スコア正なら「conservative」（例:
  Phe→Tyrは+3）、そうでなければ「non-conservative」（例: Phe→Aspは-3）
  ——アラインメント自体が使っているのと同じスコア行列を再利用しており、
  それと食い違う可能性のある別の物理化学的グルーピングを独自に保守する
  ことを避けている。
- **rdkitもC++もGPUも無し**: 本プロジェクトには低分子を扱う処理が一切
  無い（`dd_seqalign`は`dd_viewer`をまるごとvendorしているため、
  dd_compareが一切使わない単一受容体シーンビルダーのために`rdkit`まで
  引き連れてしまっているのとは対照的——ここでは`dd_viewer`のうち単純な
  文字列パッチとコンポーネント部分だけを絞り込んでvendorしている、
  `dd_compare/viewer3d/__init__.py`参照）。本プロジェクト自身の処理は
  RESTフェッチと配列アラインメント（どちらも軽量）で、それより重い処理は
  既にコンパイル済みのツール（`fpocket`、PyMOLのC++コア）に任せている
  ——`dd_afpocket`のMDサンプリングや`dd_docking`のドッキング実行のような、
  本プロジェクト自身の高速化すべきホットな数値ループもGPU並列化可能な
  処理も存在しない。

## 既知の制約

- **Pfam/InterProベースの自動探索は同一ファミリーの蛋白しか見つけられ
  ない**（上記「類似蛋白の自動探索」参照）——機能的に関連していても分類が
  異なる蛋白（CDK20/CDK2に対するMAKなど）には明示的なアクセッション
  リストが必要。
- **fpocketの自動検出ポケットは、側鎖ではなく主鎖原子のみで寄与している
  残基を見落とすことがある**——例えば側鎖がATPポケットから外を向いて
  いても、主鎖のN-H/C=Oがアデニンと水素結合しているキナーゼのヒンジ残基
  など。そのような残基は、文献上既知の選択性決定residueであっても、
  `--pocket-rank 1`の裏打ち残基リストに一切現れないことがありうる。
  `--pocket-rank`で別のfpocketランクのポケットを試すことはできるが、
  fpocketの幾何学的ポケット検出が選ばなかった特定の残基を強制的に比較へ
  含める方法は現状無い。
- **常にAlphaFoldモデルであり、実際のリガンド結合コンフォメーションでは
  ない**（比較そのものについては上記「設計メモ」で述べた意図的な選択）
  ——リガンド結合に伴う誘導適合（induced fit）によるポケット変化はこの
  比較の対象外である。実構造オーバーレイは表示専用。
- **実構造オーバーレイは蛋白ごとに`--pdb-max-structures`件まで**
  （デフォルト3、リガンドの種類ごとに1件）——`--pdb-scan-cap`件の最良
  解像度候補の中からリガンド結合エントリを優先するが、CDK2の512件の
  ように構造が大量にあるターゲットでも全件は走査しない。そのため走査
  範囲外にリガンド結合エントリが存在すれば見逃す可能性があり、その
  ターゲットがこれまでに共結晶化された全リガンドを表示するわけでは
  なく、選ばれた分だけである。重複判定は厳密なリガンドresnameの一致
  によるため、まれに同じ3文字のRCSBコードを共有する別々のリガンドが
  あれば、同一のものとして扱われてしまう。
